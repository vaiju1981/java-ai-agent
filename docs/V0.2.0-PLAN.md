# v0.2.0 Plan — "FinCopilot": a grounded finance copilot

> **Status:** proposal for review. No v0.2.0 code is written until this plan is approved.
> **Working name:** "FinCopilot" (placeholder — easily changed).

## 0. Principle

v0.1.x proved the framework with demos. v0.2.0 proves it with **one real product, built end to end**.
We do not add to the framework speculatively — a capability earns its place only because FinCopilot
needs it. Building the real thing is what "puts the framework through the sword": the gaps it exposes
*are* the v0.2.0 framework roadmap.

---

## 1. The product

A **grounded finance copilot** for **individuals and small businesses** — public self-signup. It fuses
the two existing demo domains into one product with two specialists behind a supervisor:

| Component | From | Does |
|---|---|---|
| **Analyst** | DataAnalyst demo | "Where did my money go?" — spending breakdowns, trends, anomalies, cashflow projections. Tool-driven queries + stats over the user's *real* transactions; structured output → charts. |
| **Advisor** (grounded) | PersonalFinance demo | "Can I afford this?", "How do I cut my burn rate?" — budgeting/cashflow guidance, RAG-grounded in (a) the user's own data and (b) a curated finance knowledge base, with disclaimers + scope guardrails. |
| **Supervisor** | framework (`SupervisorAgent`) | Routes each turn to Analyst vs Advisor. |

**"Grounded" is the core promise:** every number traces to the user's data; every recommendation cites
its basis. No invented figures, no un-sourced advice.

**Why this product:** it exercises nearly the whole framework — multi-agent routing · RAG grounding ·
tool-calling · structured output · cross-session memory · guardrails · governance/audit · streaming ·
auth · PII/privacy · cost metering. Maximum sword for one build.

## 3. Definition of "production-ready" (the bar)

The app is not done until it clears all of this:

- **Frontend** — React SPA: streaming chat, dashboards/charts, account & data management, history, settings.
- **Backend** — Spring Boot service (productionized from `production-reference`): durable multi-turn
  conversations, sessions, agent orchestration.
- **Data** — Postgres + pgvector with Flyway migrations (schema sketch in §5).
- **Ingestion** — CSV/OFX import + manual entry. (Live bank aggregation: pluggable interface, not implemented.)
- **Auth** — consumer auth (OAuth/social + email), sessions, strict per-user isolation; "business" = an org with members.
- **Privacy/PII** — encryption at rest for financial data; PII redaction in logs/traces; data export + delete; retention policy.
- **Safety** — disclaimers + scope guardrails; PII + prompt-injection guardrails; read-vs-effectful tool
  authorization; human confirmation for any effectful action.
- **Observability** — OTEL traces/metrics/structured logs + dashboards; per-user token & cost accounting;
  audit of every agent decision.
- **Cost/limits** — per-user budgets, rate limiting, model fallback.
- **Ops** — Docker Compose (React + API + Postgres); env/secret config; healthchecks; backup story; one-command up.
- **Quality** — e2e tests (UI → API → agent → DB); an agent eval suite (grounding/accuracy/safety); a load test; app CI/CD.
- **Docs** — deploy guide, operator runbook, user guide; the framework cookbook updated.

## 6. Framework workstreams (what the sword reveals)

Changes to the **framework**, each forced by a concrete product need. Each lands inside the milestone
that needs it (§8).

| ID | Forced by | Change |
|---|---|---|
| **FW‑1** Conversations/sessions | multi-turn chat | First-class durable session/conversation API in the runtime (today: stateless `/turn` + a basic store); history, resumption, pagination. |
| **FW‑2** Grounding & citations | grounded advisor | Provenance flows retrieved-context → final answer; structured output carries citations. |
| **FW‑3** Typed tool results | charts/tables | Tool results beyond `String` — a typed payload contract the UI can render (touches `ToolResult`; deprecate-don't-remove). |
| **FW‑4** HITL, productionized | confirm effectful actions | Async approval-request/resume API + approval events over the stream (today `ToolApprover` is synchronous). |
| **FW‑5** PII in the trust layer | financial data | Redaction applied to audit/traces; data-subject export/delete hooks on stores. |
| **FW‑6** Per-principal metering | consumer cost control | Usage/quota seam keyed by principal/tenant + reporting (extends budget). |
| **FW‑7** Eval: grounding + safety | trustworthy advice | Extend the eval harness: numeric accuracy, grounding, disclaimer presence, out-of-scope refusal; a dataset format. |
| **FW‑8** API curation | the 122-type surface | Stable-public vs internal split; tighter module boundaries (see §7). |
| **FW‑9** Deploy/config | self-hosted | Reusable Compose template + config/secrets conventions; extend the Spring Boot starter with production knobs. |

App-only work (in `apps/fincopilot`, not the framework): React UI, auth, ingestion, the persistence schema.

## 7. Developer impact — "deprecate, don't remove"

The chosen policy for the existing v0.1.x public API (122 public types in `agent-core`):

- **Policy:** v0.2.0 ships the curated API; anything replaced becomes
  `@Deprecated(forRemoval = true, since = "0.2.0")` with a **working adapter**, removed no earlier than
  **v0.3.0**. A v0.1.x adopter's `./gradlew build` **still compiles** — deprecation *warnings*, not errors.
- **What an adopter must do:** nothing to keep compiling; follow `MIGRATION-0.2.md` (mostly mechanical
  renames/imports) to clear warnings.
- **Expected changes (finalized during the build):**
  - `agent-core` split into a curated **stable** surface vs **internal** (122 → a deliberately smaller
    public set); introduce a zero-dep `@Internal` marker.
  - `ToolResult` gains typed payloads (String path deprecated, not removed).
  - New conversation/session + grounding/citation APIs (purely additive).
  - A few naming/consistency fixes (old names kept `@Deprecated`).
- **Guardrails on us:** a deprecation table + `MIGRATION-0.2.md`, and **japicmp in CI** to diff the
  public API so we never break it by accident.
- **New promise — a public-API stability policy:** documents what is *stable* (semver-protected), what is
  *internal* (may change anytime), and the deprecation lifecycle, so post-0.2 adopters can rely on it.

## 9. Out of scope for v0.2.0

- Live bank aggregation (Plaid/Tink/etc.) — pluggable interface only.
- Mobile apps (web-responsive only).
- Regulated/individualized investment or tax advice (explicitly excluded by policy + guardrails).
- Multi-region / HA deployment (single-stack Compose is the target).

## 10. Risks & watch-items

- **Grounding correctness** — numeric accuracy is the product's credibility; the eval suite (FW‑7) gates it.
- **Regulatory line** — "information not advice" must be enforced, not just intended (guardrails + evals + UI disclaimers).
- **Cost** — a consumer app with an LLM per turn needs the tiered-model + per-user-budget design (Appendix A) to be viable.
- **API curation scope creep** — japicmp + a fixed "stable surface" list keep FW‑8 bounded.
- **PII handling** — financial data raises the bar on encryption, redaction, and delete/export correctness.

## Appendix A — model & cost tiers

Consumer economics require tiering (exact assignments tuned during M1–M2; budgets enforced per user):

| Tier | Used for | Default model |
|---|---|---|
| Light | routing, categorization, simple Q&A | `claude-haiku-4-5` |
| Standard | most analyst/advisor turns | `claude-sonnet-4-6` |
| Deep | hard analysis, grounded advisory synthesis | `claude-opus-4-8` |

Provider via the existing `agent-anthropic` `ModelPort` (direct official SDK); other substrates remain
swappable. Per-user budget enforcement uses the framework's budget seam (extended by FW‑6).
